1 Introduction

Mappings (also known as terminology mappings, correspondences, or semantic alignments) are relevant to integrate and to compare data from diverse sources. If mappings are managed in distributed environments, there is a need to reference individual mappings with an identifier. Two kinds of mapping identifiers exist:

  • Assigned identifiers must explicitly be generated once by the source of a mapping and carried along with it.
  • Derived identifiers (also known as content-based identifiers) can be generated automatically from the content of a mapping.

This specification defines a derived identifier for mappings. One use case is the distributed review of mappings: if some mapping is reviewed, the review should apply to other mappings with same content, independent from when, where, and by whom the mapping was created.

This specification consists of a definition of mapping content and the definition of a method to derive an identifier from the content. The method does not depend on any data format or programming language.

See https://github.com/ts4nfdi/mapping-sameness-identifier/issues for issues.

2.2 Mapping

A mapping is a 4-tuple of:

  • subjects: a non-empty set of absolute IRIs
  • objects: a non-empty set of absolute IRIs
  • predicate: an absolute IRI, also known as mapping type
  • truth value: either true for affirmative mappings or false for negative mappings

The mapping is also called 1-to-1 mapping, if both subjects and objects contain one IRI each.

Notes:

  • some applications also allow empty sets. This specification does not cover these!
  • some applications don’t support negative mappings
  • some applications only support 1-to-1-mappings

Questions:

  • Limited or open list of possible predicates and how to express them, e.g. CURIE or special character such as =?

Note: The same concept may be referenced by different identifiers. Normlization of these to one preferred is out of the scope of this specification.

3 Identifier

3.1 Elements

The truth value is represented by character ! for negative mappings and the empty string for affirmative mappings. This allows for each filtering of negative mappings.

All IRIs SHOULD be based on NFC-normalized Unicode strings (see RFC 3987 section 5.3.2.4).

4.4 JSKOS example

Mappings can be expressed in JSKOS data format with its Mapping Object:

{
  "type": [
    "http://www.w3.org/2004/02/skos/core#mappingRelation",
    "http://www.w3.org/2002/07/owl#sameAs"
  ],
  "from": {
    "memberSet": [ 
      { "uri": "http://example.org/A" } 
    ]
  },
  "to": {
    "memberSet": [ 
      { "uri": "http://example.com/B" }
    ]
  }
}

Note that in this example http://www.w3.org/2004/02/skos/core#mappingRelation does not encode the mapping predicate but the fact that this JSON object is a JSKOS mapping!

JSKOS supports mapping sources and mapping objects with multiple source but negative mappings are not supported yet (see this issue).

4.5 RDF examples

A non-negative 1-to-1 mapping can be expressed in form of an RDF triple with mapping subjects being the RDF triple subject, mapping objects being the RDF triple object, and mapping predicate being the RDF triple predicate. An RDF triples can only represent a mappings if all of its elements are absolute IRIs.

Example: the RDF statement

<http://example.org/A> <http://www.w3.org/2002/07/owl#sameAs> <http://example.com/B> .

Has mapping sameness identifier …(TODO)…

Expressing negative mappings and mappings beyond 1-to-1 mappings in RDF require more complex features and conventions such as reification or triple terms (also known as RDF-star, introduced in RDF 1.2). The choice of these features to express mappings is out of the scope of this document. The combination of mapping sameness identifiers and RDF statements to express mappings beyond non-negative 1-to-1 mappings is an anti-pattern:

# Don't do this, it's a contradiction:
<some:mapping:sameness:id:of:non:negative:mapping> a ontology:NegativeMapping .
